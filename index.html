POD

当一个pod含有多个容器时，这些容器都会部署在同一个node上，而不会跨node部署。 一个容器只能跑一个进程，一个应用由多个进程组成，所以将多个互相关联的容器放在一起组成一个pod，再由多个紧密相关的pod组成一个应用。同一个pod里的容器共享同一个namespace(包括：network,UTS,IPC,PID)

描述一个pod的yaml文件主要由3个主要的内容组成：

  • metadata描述了pod的元信息，如：name，namespace等
  • spce描述了pod的的组成内容，如containers, volumes等
  • status描述了pod的当前运行状态信息，包括ip等（在创建新pod时，status是完全不存在的）

一份简单的yaml文件： ` apiVersion: v1 kind: Pod metadata: name: nginx-manual spec: containers:

  • image: nginx:1.8 name: nginx ports:
    • containerPort: 80 protocol: TCP `

k8s学习笔记(二) - Docker和k8s使用笔记

镜像相关

  • 镜像的构建并不是由Docker client执行的，在构建镜像时，在构建目录下的文件都会上传到Docker daemon中，由Docker daemon来构建，所以不要在构建目录中放不相关的文件，否则会拖慢构建的进度。
  • 镜像在下载时会单独下载每一个layer，并且只下载机器上并不存在的layer
  • Dockerfile中的每一行命令都是一个layer

k8s命令

将kubectl设置别名，在$HOME/.bashrc中加入下面语句 alias k=kubectl

查看node kubectl get nodes

查看具体的node kubectl describe node {nodename}

运行image kubectl run hello --image=nginx --port=80

因为k8s以pod为调度单位，所以并不存在将容器列出来的命令，只有将pod列出来的命令： kubectl get pods

列出pod的更详细信息： kubectl get pods -o wide

查看具体的pod kubectl describe pod {podname}

创建LoadBalancer的Service： kubectl expose rc {podname} --type=LoadBalancer --name {servicename}

查看service： kubectl get services

查看deployment： kubectl get deployment

水平扩展deployment： kubectl scale deployment {deployment-name} --replicas=3

POD

查看某个pod的yaml文件： kubectl get pod {pod-name} -o yaml

根据yaml文件创建pod： kubectl create -f {filename}.yaml

使用k8s记录pod的日志： kubectl logs {pod-name}

如果一个pod里含有多个container，就要指定记录日志的container： kubectl logs {pod-name} -c {con-name}

k8s创建容器的过程

在k8s中创建容器的过程如下所示：

  • 在本地机器构建好镜像后，推送到镜像中心
  • 在本地机器执行kubectl run命令
  • 本地机器调用k8s的master node上的API Server，创建一个Deployment
  • Deployment创建一个新的pod，由Scheduler调度到worker node上
  • worker node上的kubelet看到新来的pod后就调用docker命令从镜像中心拉取镜像并运行

k8s学习笔记(一) - k8s背景介绍

发表于 2020-12-11 | 更新于: 2020-12-14 | 分类于 k8s学习

为什么需要k8s

  • 隔离不同的应用，虽然可以用虚拟机的方案，但是虚拟机会消耗更多的硬件资源，而容器则是轻量级的隔离方案，除了进程本身之外，不会消耗额外的资源，从而运行更多的应用。但存在着一个问题：容器因为完全共享了同一个os，所以并不能互相彻底的隔离。
  • 服务运行时对资源的动态调整
  • 为应用提供一个一致的运行环境，将Devops（开发人员于运维人员全程参与到项目的研发，部署和运营过程，可以让开发人员根据用户反馈快速迭代产品，同时也能了解到运维人员遇到的问题）转换为NoOps（开发人员不需要了解硬件的底层知识，同时也不需要与运维团队打交道）

如何实现隔离

两个重要概念：

  • Linux Namespaces，使得每个进程只能看到自己的资源（文件，网络等），Linux的进程在运行时，只能看到同属于一个namespace的资源，Linux的namespace一共有以下几种：
    • Mount(mnt)
    • Process ID(pid)
    • Network(net，应用运行在哪个网卡上，每个网卡只能属于一个namespace，但是可以从一个ns挪到另一个ns上)
    • Inter-process communication(ipc)
    • UTS(隔离hostname和domain name，使得进程看到不同的hostname，从而认为自己运行在不同的机器上)
    • User ID(user)
  • Linux Control Groups(cgroups)，限制进程可以使用的资源（CPU，内存，带宽等）

不同的APP如何共享同样的文件

镜像是由不同的分层（layers）组成，不同的镜像可能使用完全一致的layer，写时复制的功能？

容器仍然存在以下问题

  • 如果机器不能给容器提供同样的内核模块，那么容器就运行不起来
  • 在x86上构建的镜像也不能运行在ARM的机器上

k8s集群包含的组件

k8s集群由master node和worker node组成，其中master mode运行着k8s的控制平面，用于保存和控制集群的状态，包含了以下组件：

  • K8s API Server，与其他控制平面组件通信
  • Scheduler，调度你的应用到worker node上运行
  • Controller Manager，执行集群操作，譬如复制组件，跟踪worker node，处理node failure
  • Etcd，一个高可用的分布式数据库，用于保存集群配置

worker node是用于运行应用的实际机器，主要包含了以下组件：

  • Docker，rkt或其他容器运行环境
  • Kubelet，与master的API Server通信并控制容器
  • Kube-proxy，负载均衡

容器漂移

如果一个应用由多个相同的容器共同提供服务，那么这些容器都会绑定到一个固定的静态IP上，由kube-proxy提供负载均衡的能力，即使这些容器发生了漂移，转移到了不同ip的机器上，也不会影响到该应用对外提供的服务。

Linux下的rawsocket

发表于 2019-11-02 | 更新于: 2019-11-16

数据包在协议栈内的处理流程

如下图所示，网卡接收到了数据包，在链路层处理完之后会产生两条处理路径：

  • 第一条路径是继续送到上层的IP和TCP层进行处理，然后送到一般的socket接口（譬如stream（TCP)和datagram(UDP) ），此时数据包已经完全去掉了所有协议层的数据包头信息（IP地址，MAC地址等），只剩下了数据负载（payload）；使用stream或datagram类型的socket时，需要指定IP和端口信息，就是为了在IP和TCP层处理数据包时捕获那些满足要求的数据包；
  • 另一条路径是绕过TCP/IP层的处理，直接送到raw socket（raw socket捕捉到的数据是内核协议栈中数据包的一份拷贝），从而使得应用层可以直接访问完全没有处理过的数据包，从而获取数据包里的包头信息（IP地址，MAC地址等）；注意的是，当网卡设置为“混杂”模式时才会捕获全部到达网卡的数据包，如果设置为“非混杂”模式，则只会捕获目的地址为本机的数据包。

网卡对数据帧进行硬过滤（数据链路层过滤）

首先进行数据链路层校验和处理，如果校验和出错，直接扔掉，而不会拷贝给sock_raw套接字；然后根据网卡的模式不同会有不同的动作：如果设置了promisc混杂模式的话，则不做任何过滤直接交给下一层输入例程，否则非本机mac或者广播mac会被直接丢弃。

向上层递交数据链路层数据帧

在进入网络层之前，系统会检查系统中是否有通过socket(AF_PACKET, SOCK_RAW, ...)创建的套接字；如果有并且与指定的协议相符的话，系统就给每个这样的socket接收缓冲区发送一个数据帧拷贝。然后进入网络层。

网络层过滤

IP层会对该数据包进行软过滤：检查校验或者丢弃非本机IP或者广播IP的数据包等

向上层递交网络层数据包

在进入运输层（如TCP、UDP例程）之前，系统会检查系统中是否有通过socket(AF_INET, SOCK_RAW, ...)创建的套接字；如果有的话并且协议相符，系统就给每个这样的socket接收缓冲区发送一个数据包拷贝（不管在任何情况下，永远都包含IP数据包头）。然后进入运输层。

运输层处理数据包

这一步由系统内核来处理运输层的协议，用户层无法干涉。处理完之后，如果合法，将向用户层递交数据，进而进入用户层处理。

进入用户层

这一层在用户态，由用户应用程序来完成。

编写raw socket

raw socket的函数原型是：socket(int socket_family, int socket_type, int protocol);

一共有三种方式来创建raw socket

下面分别讨论前两种方式的raw socket：

  • 第一种，socket(AF_INET, SOCK_RAW, IPPROTO_XXX)；这个socket比较适合写ping程序

    • 这个SOCKET接收的数据是包含IP头的IP数据包（不管protocol是任何协议），并且是已经重组好的IP包（协议类型为IPPROTO_XXX且发往本机的IP数据包）。
    • 注意：不能捕获目的IP非本机的数据包（在网络层会被过滤掉），也不能捕获从本机发送的数据包
    • 如果protocol是IPPROTO_RAW(255)，这时候，这个socket只能用来发送IP包，而不能接收任何的数据。发送的数据需要自己填充IP包头，并且自己计算校验和
    • 对于protocol为0（IPPROTO_IP)的raw socket。用于接收任何的IP数据包。其中的校验和和协议分析由程序自己完成。
    • 如果protocol既不是0也不是255，那么sock_raw既可以接收数据包，也可以发送数据包。
    • 如果 raw socket 没有使用 connect 函数绑定对方地址时，则应使用 sendto 或 sendmsg 函数来发送数据包，在函数参数中指定对方地址。如果使用了 connect 函数，则可以直接使用 send、write 或 writev 函数来发送数据包。
    • 如果 raw socket 调用了 bind 函数，则发送数据包的源 IP 地址将是 bind 函数指定的地址。否则，内核将以发接口的主 IP 地址填充；如果设置了 IP_HDRINCL 选项，则必须手工填充每个发送数据包的源 IP 地址。

  • 第二种，socket(PF_PACKET, SOCK_RAW, htons(ETH_P_XXX))；

    • 这种套接字可以监听网卡上的所有数据帧，可以接收三种类型的数据帧：
      • 发往本地mac的数据帧
      • 从本机发送出去的数据帧(第3个参数需要设置为 ETH_P_ALL)
      • 非发往本地mac的数据帧(网卡需要设置为 promisc 混杂模式)
    • 协议类型一共有四个：
      • ETH_P_IP 0x800 只接收发往本机mac的IP类型的数据帧
      • ETH_P_ARP 0x806 只接收发往本机mac的ARP类型的数据帧
      • ETH_P_ARP 0x8035 只接收发往本机mac的RARP类型的数据帧
      • ETH_P_ALL 0x3 接收发往本机mac的所有类型（IP、ARP、RARP）的数据帧，并接收从本机发出的所有类型的数据帧（混杂模式打开的情况下，会接收到非发往本地mac的数据帧）
    • 发送数据帧，需要自己组织整个以太网数据帧，所有相关的地址使用struct sockaddr_ll,而不是struct sockaddr_in（因为协议簇是PF_PACKET，不是AF_INET了）。

netlink

发表于 2019-09-14 | 更新于: 2019-09-30

与内核常用的通信方式

用户态程序与内核态程序进行通信的方式主要有以下几种：

  • /proc虚拟文件系统，/proc是linux的一个目录，该目录下存储了内核的运行信息，很多linux查询命令其实都是输出/proc目录下的特定信息。当内核的某个变量保存到了/proc目录下之后，用户可以直接修改该变量的值，就可以改变内核的某个模块的运行逻辑，从而实现用户态与内核态程序通信的目的。
  • netlink通信，netlink通信的过程其实和socket通信一样，在socket通信中，客户端负责封装消息体，然后通过socket机制把消息发送到服务端，服务端则负责解读消息体并执行消息处理程序。由于socket通信是全双工的，所以当用户态程序与内核态程序进行通信时，客户端既可以发送消息，也可以接收内核态返回的消息，反之同理。netlink不仅可以实现用户-内核空间的通信还可以实现用户空间两个进程之间，或内核空间两个进程之间的通信。

netlink简介

在编写netlink程序时，程序员需要自行封装消息体，同时需要自定义处理消息的逻辑。一般的做法是编写一个LKM（Linux Kernel Module，是一个后缀为ko的文件），在LKM里定义内核处理netlink消息的逻辑，然后把ko文件加载到内核中，就可以实现用户态和内核态程序之间的通信。netlink支持两种类型的通信方式：单播和多播，单播指的是一个用户进程和一个内核子系统之间进行1对1的通信；多播指的是一个内核子系统和多个用户进程之间进行1：N的通信。本文讨论的是单播通信方式，那么第一个问题来了，当用户态程序通过netlink往内核发送消息时，用户态是如何标识这条消息发给哪个内核子系统的呢，当内核子系统通过netlink往用户态程序发送消息时，用户态程序是如何知道这是发给自己的消息呢？答案在下面的数据结构里。

消息头数据结构

消息头由结构体nlmsghdr定义

结构体的字段含义如下：

  • nlmsg_len：整个消息的长度，按字节计算。包括了Netlink消息头本身。
  • nlmsg_type：消息的类型，内核通过这个字段来判断这个消息的用途，用户可以自定义新的消息类型(定义的值不能小于0x10)，也可以使用已经定义好的类型，如下：
    • NLMSG_NOOP：空消息，什么也不做；
    • NLMSG_ERROR：指明该消息中包含一个错误；
    • NLMSG_DONE：如果内核通过Netlink队列返回了多个消息，那么队列的最后一条消息的类型为NLMSG_DONE。
    • NLMSG_OVERRUN：暂时没用到。
    • 假如现在要允许内核读写特定的网络设备，通常的做法是定义两种消息类型：
      • #define MSG_SETCFG 0x11
      • #define MSG_GETCFG 0x12
  • nlmsg_flags：附加在消息上的额外说明信息，其取值包括以下内容：

    • #define NLM_F_REQUEST 1 // Message is a request

    • #define NLM_F_MULTI 2 // Multipart message

    • #define NLM_F_ACK 4 // ACK message requested

    • #define NLM_F_ECHO 8 // Request to echo the request

    • #define NLM_F_ROOT 0x100 //Return based on root of tree.

    • #define NLM_F_MATCH 0x200 // Return all matching entries.

    • #define NLM_F_ATOMIC 0x400 // Obsoleted, once used to request an atomic operation.

    • #define NLM_F_DUMP (NLM_F_ROOT|NLM_F_MATCH) //Return a list of all objects (NLM_F_ROOT|NLM_F_MATCH).

    • #define NLM_F_REPLACE 0x100 // Replace an existing object if it exists

    • #define NLM_F_EXCL 0x200 // Do not update object if it exists already

    • #define NLM_F_CREATE 0x400 // Create object if it does not exist yet

    • #define NLM_F_APPEND 0x800 // Add object at end of list

  • nlmsg_seq：消息序列号。因为Netlink是面向数据报的，所以存在丢失数据的风险，但是netlink提供了如何确保消息不丢失的机制， 让程序开发人员根据其实际需求而实现。消息序列号一般和NLM_F_ACK类型的消息联合使用，如果用户的应用程序需要保证其发送的每条消息都成功被内核收到的话，那么它发送消息时,需要用户程序自己设置序号，内核收到该消息后对提取其中的序列号，然后在发送给用户程序 回应消息里设置同样的序列号。有点类似于TCP的响应和确认机制。
  • nlmsg_pid：当用户空间的进程和内核空间的某个子系统之间通过Netlink建立了数据交换的通道后,netlink会为每个这样的通道分配一个唯一的数字标识。其主要作用就是将来自用户空间的请求消息和响应消息进行关联。说得直白一点，假如用户空间存在多个用户进程，内核空间同样存在多个进程,netlink必须提供一种机制用于确保每一对“用户-内核”空间通信的进程之间的数据交互不会发生紊乱。通常情况下，用户空间的进程在向内核发送消息时一般通过系统调用getpid()将当前进程的进程号赋给该变量，即用户空间的进程希望得到内核的响应时才会这么做。从内核主动发送到用户空间的消息该字段都被设置为0。

常用到的几个宏：

  • NLMSG_SPACE(MAX_PAYLOAD): 该宏用于返回不小于MAX_PAYLOAD且4字节对齐的最小长度值，一般用于向内存系统申请空间是指定所申请的内存字节数,所申请的空间里不包含netlink消息头部所占的字节数

3.创建并填充sk_buff

sk_buff,全称socket buffers，简称skb，中文名字叫套接字缓存。它作为网络数据包的存放地点，使得协议栈中每个层都可以对数据进行操作，从而实现了数据包自底向上的传递。该结构维护一个收到的或者要发送的网络包。但其本身并不包含存放网络包的数据的存储区。存储区是另外单独分配的内存空间，但该结构说明了如何访问存储区空间，如何维护多个存储区空间以及存储网络包解析的成果。

网络设备

中继器

+ 协议栈中的位置：物理层
+ 作用：对网线上的信号进行中继和放大，实现信号在网线上的远距离传输

集线器

协议栈中的位置：物理层
作用：中继器只有两个接口，而集线器就是一个多口中继器
弊端：从任意接口收到的数据，都会往其他所有接口泛洪，因为集线器不能识别数据包的寻址信息和上层内容，无法对终端主机进行隔离，使得所有的主机都处于同一个冲突域中，导致网络带宽利用率低

网桥

协议栈中的位置：链路层
改进对象：集线器的泛洪问题
作用：记录终端主机的MAC地址并生成MAC表，网桥根据MAC表转发主机之间的数据流；因此能够隔离冲突域( ![](网络设备/bridge.png)，例如PC1发往PC2的数据不会转发到E2接口 )
弊端：接口有限，默认只有两个接口，可隔离的冲突域数量有限

交换机

协议栈中的位置：链路层
功能：记录终端主机的MAC地址并生成MAC表，交换机根据MAC表转发主机之间的数据流；
改进：交换机是在网桥的基础上进行延伸和升级，相比于网桥，有以下几个优势：
- 接口数量更加密集，每个主机都在独立的冲突域
- 使用ASIC硬件芯片进行高速转发
- 能够进行VLAN隔离（不仅可以隔离冲突域，还可以通过VLAN隔离广播域）
交换机是局域网设备，无法实现远程广域网通信

路由器

协议栈中的位置：网络层
改进：
功能：基于IP寻址，采用路由表实现数据转发；用于连接不同的局域网，实现广播域隔离

AC（Access Point，无线路由器）

协议栈中的位置：
功能：可以看做是无线功能的交换机/路由器

网络架构

家庭网络

网络拓扑：通过无线路由器提供WIFI热点接入，并提供路由器连接到外网
设备：无线路由器
所用技术：WIFI，NAT，PPPOE，DHCP

小型创业公司

网络拓扑：采用两层架构，单核心拓扑
设备：路由器，交换机，服务器
所用技术：VLAN，TRUNK，DHCP，WLAN，静态路由/OSPF，PPPOE，NAT，ACL

园区网络（大中型企业，校园网）

网络拓扑：采用 接入-汇聚-核心 三层架构，双核心网络。根据网络需求，分为用户区，对内服务区，对外服务区，管理区，互联网区等，通过核心交换机和防火墙进行连接和隔离。互联网采用多出口连接，通过路由器进行拨号和NAT，通过流控产品进行负载均衡/上网行为管理，通过防火墙进行安全隔离
设备：路由器，交换机，无线AC/AP，防火墙，负载均衡器/行为管理，服务器
技术：VLAN，TRUNK，MSTP，HSRP/VRRP，Etherchannel，WLAN，NAT，ACL，PPPOE，SNMP

数据中心/IDC

网络拓扑：采用两层架构，主要分为租户区（服务器集群），互联网区，安全管理区，其中租户区采用设备虚拟化和链路虚拟化技术，将设备处理能力和链路承载能力提高，并且在服务器区放置负载均衡器，将流量合理有效分配到固定的服务器上。互联网出口区采用路由器执行BGP和地址翻转，采用IPS/防DDOS设备防止大流量泛洪攻击，采用流控执行出口负载，采用防火墙进行安全隔离。安全管理区通过防火墙进行安全接入，通过审计，日志，入侵检测，网管等产品进行全网安全管理
设备：路由器，交换机，负载均衡器，防火墙，入侵检测/入侵防御，网络审计，防DDOS，服务器
技术：VLAN,TRUNK,vPC,IRF,LISP,Vxlan,TRILL,FarbriPath,BGP,MPLS,SNMP等
