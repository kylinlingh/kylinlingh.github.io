k8s学习笔记(三) - 各种k8s元件

发表于 2020-12-17 | 更新于: 2020-12-30 | 分类于 k8s学习 | |

POD

当一个pod含有多个容器时，这些容器都会部署在同一个node上，而不会跨node部署。 一个容器只能跑一个进程，一个应用由多个进程组成，所以将多个互相关联的容器放在一起组成一个pod，再由多个紧密相关的pod组成一个应用。同一个pod里的容器共享同一个namespace(包括：network,UTS,IPC,PID)

描述一个pod的yaml文件主要由3个主要的内容组成：

  • metadata描述了pod的元信息，如：name，namespace等
  • spce描述了pod的的组成内容，如containers, volumes等
  • status描述了pod的当前运行状态信息，包括ip等（在创建新pod时，status是完全不存在的）

一份简单的yaml文件： ` apiVersion: v1 kind: Pod metadata: name: nginx-manual spec: containers:

  • image: nginx:1.8 name: nginx ports:
    • containerPort: 80 protocol: TCP `

端口转发

当你想与pod通信，而又不想通过service的方式，k8s允许使用kubectl port-forward命令，下面的命令可以将机器的本地8888端口转发到pod的8080端口上： kubectl port-forward {pod-name} 8888:8080

label

将多个容器按照某个维度打标签(key value对)，从而可以批量地对多个容器下发命令。label selector可以从三个维度来通过label过滤目标pod：

  • 包含（或不包含）某个key的标签
  • 包含某个key和value的标签
  • 包含某个key，并且其value等于或不等于特定值

annotation

annotation不像label，并不存在annotation selector

ReplicationController

ReplicationController用来控制k8s中运行的pod数量，使其符合用户的预期，流程如下：主要是通过label selector来选择对应的pod：

ReplicationController包含了三个重要的组成部分：

  • label selector，用于选择pod
  • replica count，用户期望的pod数量
  • pod template，用来创建新的pod

一个ReplicationController的例子（不需要指定label selector，因为会从template中自动产生）： apiVersion: v1 kind: ReplicationController metadata: name: kubia spec: replicas: 3 selector: app: kubia template: metadata: labels: app: kubia spec: containers: - name: kubia image: luksa/kubia ports: - containerPort: 8080

当你需要debug一个pod的时候，可以通过修改pod的label，从而将该pod移除出ReplicationController的纳管范围之内，让ReplicationController重新起一个pod来对外服务。

升级pod的方法：修改ReplicationController的templcate后删除旧pod，从而根据新的pod template创建新的pod

ReplicaSet

Replicaset的作用与ReplicationController几乎一样，只是丰富了pod selector的方法。譬如Replicaset只需要label中有同样的key就可以做分组选择，而ReplicationController则需要key和value都一致才能做分组选择。ReplicaSet的label selector包含几个关键字：key，operator，values，其中有四个有效的operator：

  • In，label的值必须匹配values中的其中一个
  • NotIn，label的值必须不能是其中之一
  • Exists，label必须是指定之一，并且不能指定values
  • DoesNotExist，label不能是这其中之一，并且不能指定values ` selector: matchExpressions:
    • key: app
      operator: In
      values:
      • kubia
        `

DaemonSet

使得每个node上运行且运行一个特定的pod，可以用于收集日志或做kube-proxy，如下所示，只要node的标签为disk=ssd，就会自动部署一个pod上去 apiVersion: apps/v1beta2 kind: DaemonSet metadata: name: ssd-monitor spec: selector: matchLabels: app: ssd-monitor template: metadata: labels: app: ssd-monitor spec: nodeSelector: disk: ssd containers: - name: main image: luksa/ssd-monitor

Job

当pod的任务完成之后，容器并不会再次重启，而是彻底的终结 apiVersion: batch/v1 kind: Job metadata: name: batch-job spec: completions: 5 # 指定这个job的运行次数，一次任务创建并运行一个pod parallelism： 2 # 可以同时运行的pod数量 template: metadata: labels: app: batch-job spec: restartPolicy: OnFailure # 当pod的进程执行完毕后，k8s的重启策略，默认为Always，一般只能该为OnFailure或Never containers: - name: main image: luksa/batch-job

CronJob

周期性任务 apiVersion: batch/v1beta1 kind: CronJob metadata: name: batch-job-every-fifteen-minutes spec: schedule: "0,15,30,45 * * * *" jobTemplate: spec: template: metadata: labels: app: periodic-batch-job spec: restartPolicy: OnFailure containers: - name: main image: luksa/batch-job

Service

` apiVersion: v1 kind: Service metadata: name: kubia spec: ports:

  • name: http # 当一次性暴露多个端口时，需要指定name
    port: 80 #对外暴露80端口
    targetPort: 8080 # 将每个链接路由到其中一个pod的8080端口上
  • name: https
    port: 443
    targetPort: 8443
    selector: app: kubia ` 往集群里的service发送请求的几种方法：

一个service的ip可以通过curl探活，但是不能通过ping来探活，因为service的集群ip是一个虚拟ip，必须和端口结合才能使用

将一个服务暴露到公网，大概有三种方法：

  • 将service type设置为NodePort，每一个物理机都会打开一个端口，然后将端口收到的流量重定向到第层服务中
  • 将service type设置为LoadBalancer，是NodePort的扩展版，提供四层的负载均衡
  • 创建一个Ingress资源，可以将多个不同的服务绑定到同一个IP上，工作于7层

NodePort

从下图可以看到NodePort的工作模式，NodePort绑定了node的30123端口，当外部请求落在node的30123端口上之后，该node会转发请求到service上，由service再选择一个pod来响应。 ` apiVersion: v1 kind: Service metadata: name: kubia-nodeport spec: type: NodePort # 指定该service的类型是NodePort ports:

  • port: 80 targetPort: 8080 nodePort: 30123 selector: app: kubia `

LoadBalancer

从下图可以看到LoadBalancer的工作模式，外部请求先落到LoadBalancer的80端口上，然后由LoadBalancer随机选择一个node来转发请求，再由service将请求随机落到不同的pod上。 ` apiVersion: v1 kind: Service metadata: name: kubia-loadbalancer spec: type: LoadBalancer ports:

  • port: 80 targetPort: 8080 selector: app: kubia `

Deployment

如果手动创建了一个pod，并且该pod被调度到node上运行，kubelet就会保证这个pod一直运行下去，即使该pod的进程崩溃，kubelet也会自动重启这个pod，因此k8s里运行的app可以自愈。但是存在一个问题，如果该pod的进程出现了bug，不能正常响应（譬如出现了死锁），k8s并不能主动重启该pod。k8s可以通过心跳检测来检查pod的进程，有三种检测方法：

  • HTTP Get请求，返回值为2xx或3xx就正常，否则就重启容器
  • TCP socket，只要能成功建立socket连接就说明正常，否则就重启容器
  • 在容器内部执行Exec调用，检查执行的结果值是否为0

定义一个通过http请求做心跳检测的pod ` apiVersion: v1 kind: Pod metadata: name: kubia-liveness spec: containers:

  • image: luksa/kubia-unhealthy name: kubia livenessProbe: httpGet: path: / port: 8080 initialDelaySeconds: 15 # 一定要设置该值，否则在pod启动后就马上进行探测，而pod由于还未就绪就会响应失败，从而导致pod被重启 `
