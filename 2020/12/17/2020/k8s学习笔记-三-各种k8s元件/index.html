POD

当一个pod含有多个容器时，这些容器都会部署在同一个node上，而不会跨node部署。 一个容器只能跑一个进程，一个应用由多个进程组成，所以将多个互相关联的容器放在一起组成一个pod，再由多个紧密相关的pod组成一个应用。同一个pod里的容器共享同一个namespace(包括：network,UTS,IPC,PID)

描述一个pod的yaml文件主要由3个主要的内容组成：

  • metadata描述了pod的元信息，如：name，namespace等
  • spce描述了pod的的组成内容，如containers, volumes等
  • status描述了pod的当前运行状态信息，包括ip等（在创建新pod时，status是完全不存在的）

端口转发

当你想与pod通信，而又不想通过service的方式，k8s允许使用kubectl port-forward命令，下面的命令可以将机器的本地8888端口转发到pod的8080端口上： kubectl port-forward {pod-name} 8888:8080

label

将多个容器按照某个维度打标签(key value对)，从而可以批量地对多个容器下发命令。label selector可以从三个维度来通过label过滤目标pod：

  • 包含（或不包含）某个key的标签
  • 包含某个key和value的标签
  • 包含某个key，并且其value等于或不等于特定值

annotation

annotation不像label，并不存在annotation selector

ReplicaSet

Replicaset的作用与ReplicationController几乎一样，只是丰富了pod selector的方法。譬如Replicaset只需要label中有同样的key就可以做分组选择，而ReplicationController则需要key和value都一致才能做分组选择。ReplicaSet的label selector包含几个关键字：key，operator，values，其中有四个有效的operator：

  • In，label的值必须匹配values中的其中一个
  • NotIn，label的值必须不能是其中之一
  • Exists，label必须是指定之一，并且不能指定values
  • DoesNotExist，label不能是这其中之一，并且不能指定values

Service

往集群里的service发送请求的几种方法：

一个service的ip可以通过curl探活，但是不能通过ping来探活，因为service的集群ip是一个虚拟ip，必须和端口结合才能使用

将一个服务暴露到公网，大概有三种方法：

  • 将service type设置为NodePort，每一个物理机都会打开一个端口，然后将端口收到的流量重定向到第层服务中
  • 将service type设置为LoadBalancer，是NodePort的扩展版，提供四层的负载均衡
  • 创建一个Ingress资源，可以将多个不同的服务绑定到同一个IP上，工作于7层

PersistentVolumes

为了屏蔽底层的volume细节，可以使用PersistentVolumes和PersistentVolumeClaims。如下图所示，由k8s集群的管理员先设置底层的存储细节（容量大小和访问方法）并注册一个PersistentVolume到k8s API服务器上。当用户需要使用持久化存储时，就会创建一个PersistentVolumeClaim，并制定最小存储空间和访问模式，然后往k8s API server里提交 PVC，此时k8s就会找到对应的PV，然后将其绑定到PVC上。其他用户不能使用同一个PV，除非PVC已经和PV解绑。

ConfigMap

配置应用可以通过以下三种方法：

  • 通过命令行传递参数到容器中
  • 在容器里设置环境变量
  • 通过特殊卷挂载配置文件

ConfigMap是一个独立的对象，里面存储了key value对，一个应用可以直接读取ConfigMap对象里的配置信息，也可以通过容器的环境变量来加载ConfigMap的配置信息，把不同的ConfigMap放在不同的namespace里即可实现不同环境加载不同配置文件的功能。

使用环境变量或者命令行参数来传递配置信息的最大缺点是容器运行时无法更新配置信息，而通过挂载卷的方式使用ConfigMap则可以做到热更新，因为引用了ConfigMap的卷后，容器里的配置文件会同步更新，k8s也可以在文件更新后往容器发送对应的信号。

Secret

Secret类似于ConfigMap，只不过一般用于保存证书，常见的场景是nginx接受https协议。每个容器会自动挂载一个名为default-token的Secret的卷，用于与k8s API server通信。

Deployment

如果手动创建了一个pod，并且该pod被调度到node上运行，kubelet就会保证这个pod一直运行下去，即使该pod的进程崩溃，kubelet也会自动重启这个pod，因此k8s里运行的app可以自愈。但是存在一个问题，如果该pod的进程出现了bug，不能正常响应（譬如出现了死锁），k8s并不能主动重启该pod。k8s可以通过心跳检测来检查pod的进程，有三种检测方法：

  • HTTP Get请求，返回值为2xx或3xx就正常，否则就重启容器
  • TCP socket，只要能成功建立socket连接就说明正常，否则就重启容器
  • 在容器内部执行Exec调用，检查执行的结果值是否为0
